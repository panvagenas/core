# add_classes_dir( string $classes_dir )

Adds a new classes directory.

Adds a new classes directory.

Parameters

$classes_dir
string

$classes_dir Any classes directory; w/ basename classes.

Throws

exception

If invalid types are passed through arguments list.


exception

If $classes_dir is empty, or is NOT an existing directory.


exception

If $classes_dir does NOT have a classes basename.


Note

This directory MUST use sub-directories with a namespace/class hierarchy. Nested sub-namespaces are also supported in the same structure (i.e. /namespace/sub_namespace/class). Underscores in any namespace and/or class should be replaced with dashes in the file structure.


Each classes directory is pushed to the top of the stack by this routine. This way load_ns_class() is looking for class files in the last directory first; and then searching up the stack until it finds a matching class file.

This makes it possible for pro add-ons and/or other plugins integrated with the core, to create classes that will override any that may already exist for a particular plugin. For example, a pro add-on MAY wish to override default/free plugin classes.

However, the underlying core classes (when called explicitly); may NOT be overwritten. We keep the core classes directory ALWAYS at the top of the stack when adding new directories.


A plugin CAN override core classes (for use in that plugin); by extending core classes and placing them into the plugin's own classes directory. That is NOT handled here though; it is handled with © dynamic class instances. See: websharks_core_v000000_dev\framework::__get(), websharks_core_v000000_dev\framework::__call()
